## 1. Введение

### 1.1. Зачем двухагентная схема

Платформа базы знаний reHome разрабатывается AI-агентами Claude Code. Один и тот же агент-разработчик не может объективно оценить собственную работу: он подвержен confirmation bias, склонен «доделать чуть позже», может не заметить нарушения собственного плана. Двухагентная схема (Разработчик + независимый Проверяющий) решает эти проблемы:

- Разделение труда: один пишет — другой проверяет. У каждого роль и зона ответственности.

- Объективность: Проверяющий не имеет инвестиции в конкретное решение и судит по критериям ТЗ.

- Трассируемость: каждое решение проходит ревью, фиксируется в PR, остаётся в истории.

- Защита от регрессий: правила раздела 5 (запрет костылей, защита закоммиченного кода) исполняются автоматически.

- Эскалация к человеку-Архитектору только в спорных случаях — это экономит время и снижает риск ошибок при автономной работе.

### 1.2. Область применения

- Этот документ применяется ко всем коммитам в защищённые ветки репозиториев платформы reHome после даты введения.

- Существующий код проекта (на момент введения ТЗ) — это baseline, не подлежащий ретроактивной проверке.

- В рамках первой итерации применяется к новым репозиториям компонентов базы знаний: wiki, vault (менеджер паролей), files (хранилище документов), kb-search (RAG-движок), staff (админка).

- При успешной обкатке схема распространяется на основной репозиторий платформы reHome (бэкенд Django, фронтенд Next.js).

### 1.3. Допущения

- Разработка ведётся преимущественно AI-агентами Claude Code под наблюдением человека-Архитектора (директор / технический руководитель).

- Архитектор имеет право решающего голоса по любому спорному вопросу.

- Проект разворачивается итеративно (см. Roadmap в основных ТЗ) — не «всё и сразу».

- Существующий код в репозиториях до введения ТЗ — это контекст, а не догма; он может быть переиспользован, переработан или замещён по результатам Phase 0.

## 2. Принципы разработки — жёсткие правила


> **📌 Высший приоритет**
>
> Правила этого раздела имеют высший приоритет. Их нарушение блокирует merge независимо от технического качества кода. Проверяющий обязан отклонить PR при любом нарушении правил данного раздела. Отступление допустимо только через явное согласование Архитектора.


### 2.1. Двухагентный процесс — обязателен

Любая задача, результатом которой является изменение защищённой ветки репозитория, проходит через два независимых AI-агента: Разработчика и Проверяющего.

- **Запрещено:** Разработчик коммитит самостоятельно, минуя Проверяющего.

- **Запрещено:** Проверяющий пишет код за Разработчика — он только проверяет.

- **Запрещено:** Использование одного агентского контекста для обеих ролей в рамках одной задачи.

- **Допустимо:** Архитектор может выполнить любую роль вручную в случае эскалации.

### 2.2. Запрет на костыли

Разработчик обязан решать поставленную задачу по существу. Костыли — временные или субоптимальные обходы, скрывающие проблему вместо её устранения — запрещены. Полный операционный список — раздел 5.2.

Если Разработчик считает, что без временного решения не обойтись, он обязан:

1.  Явно остановиться и сообщить Проверяющему/Архитектору о ситуации.

2.  Предложить корректное решение и оценить его трудоёмкость.

3.  Получить явное согласование на временное решение, если корректное признано слишком дорогим прямо сейчас.

4.  Зафиксировать временное решение как технический долг (TODO с tracking-номером и сроком).


> **⚠️ Важно**
>
> «Получил согласование» означает явное письменное «да» от Архитектора в комментарии к задаче или PR. Молчаливого согласия или предположения «он бы согласился» — недостаточно.


### 2.3. Запрет на изменение закоммиченного кода без согласия

После того как код попал в защищённую ветку через одобренный PR, он считается стабильной частью проекта. Изменения в нём допустимы только в следующих случаях:

- Найден баг, требующий правки — оформляется как отдельный PR с явной отметкой «fix», ревьюится по полной процедуре.

- Архитектор явно одобрил рефакторинг — указывается коммит/файлы и причина.

- Изменение требуется для выполнения новой задачи, и Проверяющий явно одобрил его в рамках текущего PR.

- **Запрещено:** Менять, переписывать или удалять чужой ранее закоммиченный код «по дороге» (drive-by changes) без отдельного согласования.

- **Запрещено:** Использовать git commit --amend или git rebase -i на коммитах, уже попавших в защищённую ветку.

- **Запрещено:** force-push в защищённые ветки в любых обстоятельствах.

### 2.4. Прозрачность и трассируемость

Каждое действие в репозитории должно быть восстановимо по артефактам:

- Каждый PR ссылается на конкретное FR из ПЗ либо на конкретный пункт ТЗ.

- Каждый коммит имеет внятное сообщение в формате Conventional Commits (см. раздел 6.2).

- Каждое архитектурное решение фиксируется в ADR в директории /docs/adr.

- Каждое отступление от правила фиксируется в комментарии PR с обоснованием и подтверждением Архитектора.

### 2.5. Принципы, специфичные для reHome

#### 2.5.1. Минимум внешних сервисов

Принцип reHome: разрабатываем сами, внешние сервисы — только критически необходимый минимум (см. документ «База знаний v1.2», раздел 1.4). При появлении новой задачи Разработчик в плане обязан явно ответить на вопрос: можно ли решить задачу с использованием уже существующего самописного кода или open-source self-hosted компонентов? Подключение нового внешнего SaaS-сервиса — отдельное архитектурное решение, требует ADR и согласования Архитектора.

#### 2.5.2. Соответствие ФЗ-152 по умолчанию

Любой код, обрабатывающий персональные данные пользователей, обязан соответствовать требованиям ФЗ-152 (см. документ «База знаний v1.2», раздел 4). Это означает:

- Все ПДн шифруются в покое (AES-256) и в передаче (TLS 1.3).

- Логирование операций с ПДн — обязательно: actor, action, timestamp, IP, related_entity.

- Ролевая модель доступа (RBAC) — проверка прав на каждое чтение и каждую запись.

- Данные хранятся на серверах в РФ.

- Передача ПДн третьим лицам — только при наличии согласия пользователя на конкретного получателя.

Проверяющий обязан при ревью кода, обрабатывающего ПДн, отдельно проверить эти пункты. Без согласования Архитектора код, нарушающий хотя бы одно требование, не попадает в защищённые ветки.

#### 2.5.3. Двухконтурность данных — критическая инвариант

Базы знаний reHome разделены на публичный и внутренний контуры (см. документ «База знаний v1.2», раздел 1.2). Это разделение должно быть инвариантом архитектуры:

- В моделях данных у каждой статьи/чанка/документа есть поле access_level.

- Фильтрация по access_level применяется на уровне хранилища (Qdrant payload filter, Postgres WHERE), а не на уровне приложения.

- scope-параметр запроса (см. документ «Чат-поиск ТЗ v2», раздел 2.2) определяется бэкендом из сессии пользователя, не передаётся клиентом.

- Любой PR, потенциально влияющий на разделение контуров, требует security-теста: гость / арендатор / собственник пытаются получить контент с access_level выше своих прав — все попытки должны быть отклонены.

## 3. Роли агентов

### 3.1. Агент-Разработчик

Запускается как сессия Claude Code с CLAUDE.md разработчика (см. Приложение A). Цель: реализовать одну задачу за один цикл.

#### 3.1.1. Обязанности

5.  Прочитать задачу, ПЗ (документы «База знаний v1.2», «Чат-поиск ТЗ v2») и релевантные ADR.

6.  Прочитать State of Code Report (артефакт Phase 0), чтобы понимать существующий контекст.

7.  Составить План разработчика (см. раздел 4.1) до написания первой строки кода.

8.  Получить одобрение плана у Проверяющего перед началом реализации.

9.  Реализовать задачу строго по утверждённому плану. Отклонения от плана требуют отдельного согласования.

10. Написать тесты до или одновременно с кодом (TDD предпочтительно, не обязательно).

11. Прогнать локально: lint, type-check, unit-tests, integration-tests.

12. Создать PR с заполненным шаблоном (см. Приложение C).

13. Реагировать на замечания Проверяющего.

#### 3.1.2. Ограничения

- НЕ имеет прав на merge в защищённые ветки.

- НЕ имеет прав на force-push, rebase или amend закоммиченного кода.

- НЕ может менять «попутно» код, не относящийся к задаче, даже если видит «явный недостаток».

- НЕ может отключать или пропускать тесты — даже временно — без явного согласования.

- НЕ может подключать новые внешние сервисы или библиотеки без явной фиксации в плане и одобрения Проверяющим.

### 3.2. Агент-Проверяющий

Запускается как отдельная сессия Claude Code с CLAUDE.md проверяющего (см. Приложение B). Цель: убедиться, что работа Разработчика соответствует задаче, ПЗ, ТЗ и не нарушает правил.

#### 3.2.1. Обязанности

14. Прочитать задачу, План разработчика и PR.

15. Пройти по чек-листу проверки (см. Приложение D) полностью, не выборочно.

16. Принять решение по PR: approve / request-changes / reject.

17. При двухфазном коммите — провести второе ревью после правок.

18. Выполнить merge после финального approve.

19. Зафиксировать результат проверки в комментарии PR (трассируемость).

#### 3.2.2. Ограничения

- НЕ пишет код за Разработчика. Может предложить псевдокод или ссылки на документацию.

- НЕ одобряет PR, не пройдя полный чек-лист — даже если «всё выглядит нормально».

- НЕ скрывает обнаруженные нарушения правил из соображений «удобства» или скорости.

- НЕ имеет права единолично разрешить отступление от правил из раздела 2 — это компетенция Архитектора.

### 3.3. Архитектор (человек-владелец)

Не агент. Человек, который владеет проектом и принимает финальные решения. Подключается:

- При эскалации спора между Разработчиком и Проверяющим (раздел 4.4).

- При запросе на отступление от правил из раздела 2.

- При фиксации архитектурных решений (создание ADR).

- При ревью результатов целой фазы разработки.

- При принятии решения, требующего бизнес-контекста (например, по сервисному платежу, налоговой модели, юридическим формулировкам).

- В любой момент по собственной инициативе.

## 4. Процесс работы (Workflow)

### 4.1. Жизненный цикл задачи

Стандартный цикл состоит из шести этапов:

20. Постановка задачи. Архитектор (или backlog) формулирует задачу со ссылкой на конкретное FR/раздел ПЗ или пункт ТЗ. Задача оформляется как GitHub Issue (или эквивалент) с шаблоном (см. Приложение E).

21. Планирование. Разработчик читает задачу, релевантные документы и существующий код. Создаёт План разработчика, публикует его в комментарии Issue.

22. Утверждение плана. Проверяющий читает план и либо approve, либо request-changes. Без approve плана Разработчик не пишет код.

23. Реализация. Разработчик в отдельной ветке реализует задачу согласно утверждённому плану. По завершении создаёт PR.

24. Проверка. Проверяющий ревьюит PR по чек-листу. Принимает решение: однофазный или двухфазный коммит.

25. Merge. Проверяющий выполняет merge после финального approve. Ветка задачи удаляется.

#### 4.1.1. Структура Плана разработчика

План разработчика — обязательный артефакт. Минимальная структура:

- Связь с ПЗ/ТЗ: номер FR / раздел / пункт.

- Что меняется: список затрагиваемых файлов и модулей.

- Что не меняется (но рассматривалось): чтобы Проверяющий не сомневался в полноте охвата.

- Подход: краткое описание решения, ключевые архитектурные выборы, отвергнутые альтернативы.

- Внешние сервисы: какие используются (из категории C документа «База знаний v1.2», раздел 1.4); требуется ли подключать новые (если да — обоснование).

- ФЗ-152 чек-лист: затрагиваются ли ПДн; если да — какие меры предусмотрены (шифрование, журналирование, RBAC).

- Тесты: какие unit/integration тесты будут добавлены, какие edge cases покрыты.

- Риски и допущения: что может пойти не так, какие предположения о существующем коде сделаны.

- Оценка масштаба: ориентировочный размер изменений (количество файлов, строк) — для выбора между однофазным и двухфазным коммитом.

### 4.2. Однофазный коммит

Применяется только к задачам, удовлетворяющим всем нижеперечисленным критериям одновременно:

- Изменение касается ≤ 3 файлов и ≤ 100 строк кода (без учёта тестов).

- Не вносит новых публичных API, новых сущностей в доменной модели, новых зависимостей.

- Не затрагивает критические пути: KYC, платежи (номинальный счёт и сервисный платёж), ПДн, авторизация, целостность данных, фильтрация access_level в базе знаний.

- Имеет тесты, которые проходят локально и в CI.

- План разработчика был одобрен Проверяющим без замечаний.

Процедура однофазного коммита:

26. Разработчик создаёт PR.

27. Проверяющий проводит ревью по чек-листу.

28. Если всё в порядке — Проверяющий ставит approve и выполняет merge.

29. Если есть любые замечания — задача автоматически переходит в режим двухфазного коммита.

### 4.3. Двухфазный коммит

Стандартный режим. Применяется ко всем задачам, не подпадающим под критерии 4.2, а также к любой задаче, где Проверяющий обнаружил замечания при первом ревью.

Процедура:

30. Фаза 1 — Первое ревью. Разработчик создаёт PR. Проверяющий ревьюит, оставляет комментарии (request-changes). PR в статусе «ожидает правок».

31. Фаза правок. Разработчик отвечает на каждый комментарий: либо вносит правку, либо аргументирует отказ. На каждый ответ — отдельный коммит (без amend).

32. Фаза 2 — Второе ревью. Проверяющий перечитывает изменения с момента первого ревью, проверяет, что все замечания закрыты. Если нет — повторяет фазу правок (без ограничения количества раундов).

33. Финальное approve и merge. Когда все замечания закрыты, Проверяющий ставит approve и выполняет merge.


> **📌 Squash или merge?**
>
> По умолчанию используется merge с сохранением истории коммитов задачи (потому что они показывают ход обсуждения и правок). Squash допустим только если Проверяющий явно решил, что промежуточные коммиты не несут ценности — это решение фиксируется в комментарии PR.


### 4.4. Эскалация и конфликты

Сценарии, когда задача эскалируется Архитектору:

- Разработчик и Проверяющий не сходятся в третьем раунде по одному и тому же замечанию.

- Разработчик запрашивает отступление от правил раздела 2.

- Проверяющий считает, что задача поставлена некорректно и противоречит ПЗ.

- Обнаружено критическое расхождение между ПЗ, ТЗ и реальностью существующего кода.

- PR блокирован более 3 рабочих дней без прогресса.

- Задача требует решения, выходящего за компетенцию агентов: юридическая формулировка (например, по сервисному платежу), бизнес-решение (тарифы, размер комиссии), стратегический выбор (новый внешний партнёр).

Эскалация оформляется явно: комментарий в PR с тегом @architect (или соответствующим механизмом нотификации), краткое описание разногласия, позиции обеих сторон. Архитектор выносит решение, фиксирует его в комментарии — это решение становится прецедентом и при необходимости попадает в ADR.

## 5. Правила работы с кодом

### 5.1. Структура коммитов

Коммит должен быть атомарным — содержать одно логическое изменение, которое можно откатить целиком без побочных эффектов. Признаки правильного коммита:

- Все тесты проходят на каждом коммите (зелёная история).

- Сообщение коммита однозначно описывает изменение.

- В коммите нет смешения рефакторинга и новой функциональности.

- В коммите нет автогенерируемых файлов, которые должны быть в .gitignore.

- В коммите нет личных конфигов, секретов, временных файлов.

### 5.2. Определение «костыля» — операционный список

Костылём считается любой из следующих паттернов, если он не явно согласован с Архитектором:

- **Хардкод того, что должно быть конфигурируемым.** Магические числа, URL, пути, креды в коде вместо конфига или переменных окружения.

- **Закомментированный код вместо удаления.** Если код больше не нужен — он удаляется (история в git). Закомментированные блоки запрещены, кроме случаев документирования будущего восстановления — с явным TODO и сроком.

- **Отключённые тесты без объяснения.** .skip, .only, @pytest.mark.skip, закомментированные тесты — все требуют тикета и срока восстановления в комментарии.

- **Try/except, глотающий ошибки.** Перехват без логирования и без специфической обработки — запрещён. except: pass — кардинальный костыль.

- **Дублирование вместо извлечения общего.** Если один и тот же блок логики появляется в третьем месте — извлекается. Принцип «правило трёх» жёсткое.

- **Обход системы типов.** any, \# type: ignore, as unknown as X без обоснования в комментарии.

- **Жёсткая зависимость от внутренних деталей других модулей.** Импорт из приватных путей, обращение к \_private атрибутам, monkey-patching.

- **Сторонние эффекты в чистых функциях.** Запись в глобальный state, обращение к файловой системе или БД из функции, которая по контракту обязана быть чистой.

- **Отключение CI-проверок.** Любые \[skip ci\], изменения .github/workflows, .eslintrc, tsconfig.json в сторону смягчения — требуют отдельного PR с обоснованием.

- **Длинный TODO без срока и владельца.** Любой TODO/FIXME/HACK должен включать ссылку на тикет и ответственного. «TODO: позже починить» — костыль.

- **Сильная связанность через глобальное состояние.** Singleton-ы, модульные глобальные переменные — допустимы только с явным обоснованием в ADR.

- **Магические строки.** Строковые литералы, по которым происходит ветвление логики, выносятся в enum/const.

#### 5.2.1. Дополнительные костыли, специфичные для reHome

- **Утечки ПДн в логи.** Логирование паспортных данных, полных номеров карт, КЭП-сертификатов, мастер-паролей — запрещено. Только маскированные представления (последние 4 цифры, хеш).

- **Пропуск проверки access_level.** Любой запрос к базе знаний / хранилищу документов / реестру квартир обязан фильтровать по access_level. Прямые SELECT \* без WHERE access_level IN (...) — запрещены.

- **Хардкод scope или роли.** Передача scope от клиента или вычисление прав на стороне фронтенда — кардинальный костыль. Только серверная проверка по сессии.

- **Обход номинального счёта.** Любая попытка провести деньги нанимателя собственнику напрямую, минуя номинальный счёт, — нарушение ст. 860.1 ГК РФ. Запрещено даже «для тестов».

- **Mock-данные в продакшн-коде.** Hardcoded тестовые пользователи, тестовые объявления, mock-ответы внешних API — должны быть в отдельных fixture-файлах, не доступных в продакшн-сборке.


> **📌 Принцип проверки**
>
> Проверяющий, обнаружив паттерн из этого списка, обязан указать конкретный пункт. Разработчик может либо устранить, либо аргументированно запросить исключение через Архитектора.


### 5.3. Запрет на изменение закоммиченного кода

Закоммиченным считается код, попавший в защищённую ветку через одобренный PR (то есть прошедший двухагентный процесс).

**Что разрешено:**

- Изменять чужой закоммиченный код, если это часть текущей задачи и план это явно предусматривает (одобрено Проверяющим в фазе планирования).

- Исправлять баги отдельным PR с меткой «fix:».

- Рефакторить отдельным PR с меткой «refactor:» и обязательным ADR на крупные рефакторинги.

**Что запрещено:**

- «По дороге» поправить чей-то стиль, переименовать переменную, переформатировать файл.

- Удалять или менять чужие тесты под предлогом «они не релевантны».

- Изменять API публичных функций, если задача этого не требует.

- Любые операции, переписывающие историю коммитов (rebase, amend, force-push) на защищённых ветках.

### 5.4. Тесты, линт, типизация — обязательны

Каждый PR обязан удовлетворять следующим условиям до merge:

- Все unit-тесты проходят локально и в CI.

- Все integration-тесты проходят (если применимы).

- Линтер не показывает ошибок (предупреждения — на усмотрение Проверяющего, по умолчанию не блокирующие).

- Type-check проходит без ошибок (Python: mypy strict; TypeScript: tsc --strict).

- Покрытие тестами для нового кода — не ниже целевого порога: 80% для бизнес-логики, 60% для UI-компонентов. Конкретные цифры фиксируются в Phase 0.

- Новые публичные функции/классы имеют unit-тесты на основные пути (happy path + минимум 1 edge case + минимум 1 error case).

- Для security-критичного кода (KYC, платежи, access_level) — отдельный набор тестов на попытки обхода прав.

### 5.5. Документация — обязательна

Документация считается частью «готового» кода, а не отдельной задачей:

- Публичные функции, классы, модули — JSDoc/docstring с описанием, параметрами, возвращаемым значением, примерами в нетривиальных случаях.

- README модуля — для каждого нетривиального модуля, с разделами «что это», «как использовать», «как тестировать».

- ADR — для каждого архитектурного решения, влияющего на ≥ 2 модуля.

- Обновление CHANGELOG.md — для всех изменений, влияющих на пользователя или на API.

- Обновление README проекта — при изменении инструкций по запуску, тестированию, развёртыванию.

## 6. Правила работы с git

### 6.1. Branching model

Используется упрощённая модель Trunk-Based Development с короткоживущими feature-ветками:

- main — защищённая ветка, готова к деплою в любой момент. Прямой push запрещён.

- develop (опционально, по решению Архитектора в Phase 0) — интеграционная ветка для накопления изменений между релизами.

- feature/\<номер-задачи\>-\<краткое-описание\> — для новой функциональности.

- fix/\<номер-задачи\>-\<краткое-описание\> — для исправлений багов.

- refactor/\<номер-задачи\>-\<краткое-описание\> — для рефакторинга.

- docs/\<краткое-описание\> — для изменений в документации, не затрагивающих код.

Ветки задач удаляются сразу после merge. Долгоживущие ветки — только develop и release/\* (если используются).

### 6.2. Conventional Commits

Сообщения коммитов следуют формату Conventional Commits:

\<type\>(\<scope\>): \<описание\> \[опциональное тело\] \[опциональный футер: BREAKING CHANGE / Refs\]

Допустимые типы:

| **Тип**  | **Когда использовать**                                               |
|----------|----------------------------------------------------------------------|
| feat     | Новая функциональность для пользователя.                             |
| fix      | Исправление бага.                                                    |
| refactor | Изменение кода без изменения внешнего поведения.                     |
| test     | Добавление/изменение тестов.                                         |
| docs     | Изменение документации.                                              |
| chore    | Рутинные задачи: обновление зависимостей, конфигурация инструментов. |
| perf     | Изменение, улучшающее производительность.                            |
| style    | Форматирование, без изменения логики.                                |
| build    | Изменения в системе сборки или внешних зависимостях.                 |
| ci       | Изменения в CI-конфигурации.                                         |

Поле scope соответствует модулю проекта. Для reHome предлагаемый набор scope:

- kb-wiki — внутренняя wiki базы знаний

- kb-help — публичный help-центр

- kb-files — хранилище документов

- kb-vault — менеджер паролей

- kb-staff — админка реестра квартир и пользователей

- kb-hr — кадровый портал

- kb-search — RAG-движок чат-поиска

- kb-eval — eval-стенд для AI

- kb-auth — единый SSO (Keycloak / собственный модуль)

- kb-infra — общая инфраструктура, миграции, деплой

Описание — в повелительном наклонении, на русском, без точки в конце, до 72 символов. Пример: «feat(kb-search): добавить hybrid retrieval поверх Qdrant и MeiliSearch».

### 6.3. PR/MR процесс

Каждый PR должен:

- Иметь заполненный шаблон (см. Приложение C).

- Ссылаться на конкретный Issue/задачу.

- Иметь зелёный CI до запроса ревью.

- Не содержать конфликтов с целевой веткой.

- Быть атомарным — одна задача, один PR. Большие задачи разбиваются на подзадачи с отдельными PR.

Размер PR: рекомендуемый — до 400 строк изменений, целевой максимум — 800 строк. PR \> 800 строк требует обоснования и согласования с Архитектором (исключения: первичная структура нового модуля, миграции, генерируемый код).

### 6.4. Защита веток

Защищённые ветки (main, develop) настроены так, чтобы технически блокировать нарушения правил. Минимальный набор настроек:

- Запрет прямого push (только через PR).

- Обязательное прохождение CI до merge.

- Обязательное review approval (≥ 1 одобрение).

- Запрет force-push.

- Запрет удаления ветки.

- Linear history (опционально, если используется squash-merge).

- Подписанные коммиты (если включены GPG-подписи в Phase 0).


> **🔧 Технический контроль**
>
> Правила раздела 2 дублируются на уровне настроек репозитория. Это критично: словесных запретов в ТЗ недостаточно — агент может их «забыть»; настройки GitHub/Gitea/GitLab делают нарушение технически невозможным.

Результат ревью фиксируется в документе /docs/phase-reviews/phase-N.md.

## 8. Definition of Done и критерии качества

### 8.1. Definition of Done для задачи

Задача считается завершённой, когда выполнены ВСЕ следующие критерии:

- План разработчика опубликован в Issue и одобрен Проверяющим.

- Код реализует все требования из задачи и не выходит за её рамки.

- Все unit-тесты проходят, покрытие нового кода ≥ 80% (бизнес-логика) / 60% (UI).

- Все integration-тесты проходят.

- Линтер и type-check без ошибок.

- CI зелёный.

- Документация обновлена (docstring/JSDoc, README, ADR, CHANGELOG где применимо).

- Если затрагиваются ПДн — выполнены требования раздела 2.5.2 (шифрование, журналирование, RBAC).

- Если затрагивается access_level — есть security-тест на попытки обхода прав.

- PR создан с заполненным шаблоном.

- Проверяющий прошёл по чек-листу и поставил approve.

- Все комментарии Проверяющего закрыты (правкой или аргументированным отказом).

- Merge выполнен Проверяющим (не Разработчиком).

- Ветка задачи удалена.

### 8.2. Definition of Done для фазы

Фаза считается завершённой, когда:

- Все задачи фазы имеют статус Done.

- Не осталось критических багов (severity P0/P1).

- Технический долг зафиксирован в виде тикетов с владельцами.

- E2E-сценарии фазы проходят в staging-среде.

- Документация фазы готова: обновлены README, ADR, CHANGELOG.

- Архитектор провёл phase-review (см. 7.5) и зафиксировал результат.

> **📌 Главное**
>
> Двухагентная схема — это не бюрократия, а механизм повышения качества при работе AI-агентов с большой степенью автономии. Каждый артефакт (план, PR, ADR, чек-лист) служит конкретной цели: либо снижает риск ошибки агента, либо делает его работу проверяемой человеком позже. Если в ходе работы окажется, что какой-то артефакт не приносит пользы — его можно упростить через изменение ТЗ. Если окажется, что нужен новый артефакт — добавить его сюда. Сам ТЗ — живой документ.

# HANDOFF: Платформа базы знаний reHome

Этот пакет — всё, что нужно для начала разработки модуля базы знаний reHome
с помощью Claude Code по двухагентной схеме (Разработчик + Проверяющий + Архитектор).

> ✅ **Все ПЗ и ТЗ — в формате Markdown.** Claude Code читает их напрямую через `Read`,
> без необходимости в pandoc, docx2txt или других конвертерах в среде агента.
> Оригинальные .docx (для просмотра человеком) лежат в подпапках `docx_originals/`.

---

## TL;DR — что делать прямо сейчас

1. Прочитать `INDEX.md` — карта документации, ключевые сущности, открытые вопросы
2. Прочитать всю папку `01_postanovka/` (ПЗ — функциональные требования)
3. Прочитать всю папку `02_process/` (ТЗ — правила разработки)
4. Скопировать содержимое `03_repo_files/` в новый репозиторий проекта
5. Заполнить плейсхолдеры `<...>` в CLAUDE.md, CLAUDE-REVIEWER.md, ADR
6. Настроить защиту веток `main` по правилам раздела 6.4 «ТЗ Claude Code»
7. Прогнать Phase 0 — выполнить инвентаризацию существующего кода (см. ниже)
8. Только после Phase 0 — начинать первые задачи в двухагентном режиме

## Зафиксированные архитектурные решения (READ-ONLY)

Эти решения уже приняты и не подлежат пересмотру в рамках текущей фазы разработки.
Любые предложения изменить — через эскалацию к Архитектору с обоснованием.

### 1. Технологический стек (ADR-0001)
- Backend: Python 3.12+, Django 5.x, FastAPI для отдельных сервисов
- Frontend: Next.js 14+ (App Router), React 18+, TypeScript strict
- БД: PostgreSQL 16+, pgvector для эмбеддингов
- Хранилище: MinIO (S3-совместимое)
- Поиск: PostgreSQL FTS + Qdrant
- Очереди: Dramatiq + Redis
- Auth: Keycloak self-hosted (SSO)

### 2. Принцип «разрабатываем сами»
Внешние сервисы — только критически необходимый минимум.
Категории A/B/C из ПЗ «База знаний v1.4», раздел 1.4.

### 3. Финансовая модель (ADR-0002)
- Залога НЕТ
- Сервисный платёж — невозвратный, идёт платформе, вносится при заезде
- Два канала: номинальный счёт (для аренды) и расчётный счёт (для дохода)

### 4. Двухконтурность данных (ADR-0003)
- Публичный контур: access_level ∈ {PUBLIC, LOGGED, AGENT}
- Внутренний контур: STAFF, LEGAL, HR_RESTRICTED
- Фильтрация на уровне хранилища, не приложения
- scope из JWT, никогда от клиента

### 5. Коллаборанты (ADR-0004)
- Единая сущность Collaborator с 14 типами и 4 финансовыми группами
- 3 уровня портала — выбирает сам коллаборант
- УК/ТСЖ привязаны к ДОМУ, не к квартире, автоматически в NONE

## Открытые вопросы, требующие решения ДО запуска

1. **Юридические правки сервисного платежа.** Договор найма и Публичная оферта
   из `04_supporting/` написаны под модель с залогом. Их нужно переработать
   с привлечением консультанта по защите прав потребителей.
   **Это блокер MVP.**

2. **Размер сервисного платежа.** Фикс или % от аренды? Единая политика или
   сегментированная? НДС или нет (зависит от налогового режима ООО)?

3. **Первые 3-5 коллаборантов для MVP.** Рекомендация: 1 клининг (B),
   1 страховая (C), 1 тестовая УК (D), банк-партнёр (A). Кого именно?

4. **Назначение Архитектора.** Это вы лично или кто-то из команды?

## Roadmap (укрупнённо)

- **Phase 0** (2 нед) — инвентаризация, фундамент, mock-сервер
- **Phase 1** (3-4 мес) — MVP базы знаний: wiki, files, vault, staff, hr
- **Phase 2** (6-10 нед) — AI-чат (RAG), eval-стенд, выбор LLM
- **Phase 3** (4-6 мес) — внутренний чат, tool use, коллаборанты с эскроу

Внутри Phase 1 — этапы реализации API: E1-E7, ~22 недели.

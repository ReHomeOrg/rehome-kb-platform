## 1. Зафиксированные архитектурные решения

### 1.1. Два чата, один движок

Принято: разрабатываем единый RAG-движок, поверх которого работают два логически и физически разделённых чата:

| **Параметр**        | **Публичный чат**                            | **Внутренний чат**                                 |
|---------------------|----------------------------------------------|----------------------------------------------------|
| Расположение        | Виджет на rehome.one + help.rehome.one/chat  | Виджет в /staff админке + Telegram-бот             |
| Аудитория           | Гости, арендаторы, собственники, агенты      | Сотрудники reHome                                  |
| Источники данных    | Индекс PUBLIC + LOGGED (для авторизованных)  | Индекс STAFF (+ все нижестоящие)                   |
| UI                  | Дружелюбный, объясняющий, с эскалацией       | Деловой, с командами и быстрыми ссылками в админку |
| Стиль ответов       | На «вы», без жаргона                         | Профессиональный, можно ссылаться на REG-XXX-XX    |
| Доступ к ПДн        | Только своих (через tool use во второй фазе) | По матрице ролей основного документа               |
| Логирование         | Запрос + анонимизированный ответ             | Полный аудит-лог с user_id                         |
| Очерёдность запуска | Фаза 1 (P0)                                  | Фаза 2 (P1)                                        |

### 1.2. AI-провайдер выбирается по результатам экспериментов

Это ключевое решение меняет архитектуру на старте: вместо интеграции с одним провайдером строим:

- **Абстракцию LLMProvider —** интерфейс с методами generate(messages), embed(texts), token_count(text). Конкретные реализации: YandexGPTProvider, GigaChatProvider, SaigaLocalProvider, OpenAIProvider (для лабораторных тестов вне продакшна) и т.д.

- **Eval-стенд —** отдельный сервис, который прогоняет фиксированный набор тестовых вопросов через все подключённые провайдеры и считает метрики качества и стоимости (см. раздел 4).

- **Feature flag для текущего провайдера —** в продакшне работает один выбранный провайдер, но возможен мгновенный переключатель без деплоя (например, через переменную окружения или админку).

- **A/B-механизм —** после выбора основного провайдера остаётся возможность направлять X% реального трафика на альтернативу для постоянной верификации.

### 1.3. Приоритеты в порядке: скорость → контроль → стоимость → точность

Этот ранг диктует:

- MVP без перфекционизма — запустить виджет на сайте с 30-50 базовыми FAQ за 4-6 недель, не дожидаясь полного наполнения базы знаний.

- Своя разработка, не SaaS — потому что контроль выше стоимости. Готовые SaaS дают быстрый старт, но запирают в чужой инструмент. На втором месте приоритетов стоит контроль — значит, готовое решение отпадает.

- Базовая стоимость не оптимизируется на старте — выбираем умеренно дорогой стек (YandexGPT Pro по умолчанию), а уже после эксперимента переходим на оптимальное соотношение цена/качество. Точность важна, но в первой версии — не критична для запуска.

- Точность повышается итеративно — после запуска MVP начинается еженедельный цикл улучшения базы знаний по логам запросов.


> **📌 Что это значит на практике**
>
> В фазе 1 (4-6 недель) запускаем максимально простой MVP с одним временным провайдером (рекомендация: YandexGPT Pro как наиболее предсказуемый по качеству на старте). Параллельно строится eval-стенд. В фазе 1.5 (2-3 недели) проводим эксперименты на накопленных реальных запросах + размеченном эталонном наборе. По результатам — выбираем боевой провайдер и переключаем продакшн.


## 2. Архитектура единого движка

### 2.1. Принципиальная схема

Один RAG-сервис обслуживает оба чата. Различие — только в параметрах конкретного запроса:

┌─────────────────────┐ ┌─────────────────────┐ │ Публичный виджет │ │ Внутренний виджет │ │ (rehome.one) │ │ (/staff) │ └──────────┬──────────┘ └──────────┬──────────┘ │ │ └───────────────┬───────────────┘ │ ▼ ┌───────────────────────────────┐ │ Chat API (FastAPI) │ │ POST /chat │ │ { query, context, scope } │ └───────────────┬───────────────┘ │ ┌───────────────────────┼───────────────────────┐ ▼ ▼ ▼ ┌──────────────┐ ┌───────────────┐ ┌────────────────┐ │ Pre-process │ │ Retrieval │ │ Post-process │ │ (классиф., │─────▶│ (Hybrid, │──────▶│ (галлюцинации, │ │ фильтры) │ │ фильтр │ │ цитаты, лог) │ └──────────────┘ │ access_lvl) │ └────────┬───────┘ └───────┬───────┘ │ │ │ ▼ ▼ ┌──────────────┐ ┌─────────────────┐ │ Qdrant │ │ LLMProvider │ │ + Meili │ │ (абстракция) │ └──────────────┘ └────────┬────────┘ │ ┌─────────────────┼─────────────────┐ ▼ ▼ ▼ ┌────────────────┐ ┌──────────────┐ ┌─────────────┐ │ YandexGPT │ │ GigaChat │ │ Saiga │ │ Provider │ │ Provider │ │ Local │ └────────────────┘ └──────────────┘ └─────────────┘

### 2.2. Параметр scope — ключ к разделению чатов

Каждый запрос к /chat API содержит scope, определяющий контекст вызова:

| **Параметр scope** | **Описание**               | **Какие фильтры применяются**                               |
|--------------------|----------------------------|-------------------------------------------------------------|
| public_anonymous   | Гость на сайте             | access_level = PUBLIC                                       |
| public_tenant      | Авторизованный арендатор   | access_level ∈ {PUBLIC, LOGGED} + user_id фильтр            |
| public_landlord    | Авторизованный собственник | access_level ∈ {PUBLIC, LOGGED} + user_id фильтр            |
| public_agent       | Агент                      | access_level ∈ {PUBLIC, LOGGED, AGENT} + premises_id фильтр |
| staff_support      | Оператор поддержки         | access_level ∈ {PUBLIC, LOGGED, STAFF}                      |
| staff_legal        | Юрист                      | Все уровни кроме HR_RESTRICTED                              |
| staff_admin        | Администратор              | Все уровни, полный аудит-лог                                |


> **📌 Безопасность**
>
> scope определяется бэкендом исходя из сессии пользователя, а не передаётся клиентом. Запрос от анонимного гостя физически не может получить scope='staff_admin' даже при попытке подделать параметр. Это критично — иначе любая ошибка в коде на клиенте превращается в утечку.


### 2.3. LLMProvider — формальный интерфейс

Каждый провайдер реализует одинаковый интерфейс. Это позволяет менять модель одной строчкой конфигурации:

## 3. Eval-стенд — как выбираем AI-провайдер

Эксперимент по выбору провайдера — это не интуитивный «попробовали — понравилось», а отдельный сервис с воспроизводимыми тестами и метриками. Eval-стенд проектируется в фазе 1 параллельно с MVP и используется во всех последующих фазах для контроля регрессии.

### 3.1. Состав эталонного набора

Эталонный набор — это размеченные вручную пары «вопрос → ожидаемый ответ + ожидаемые источники». Минимум 200 пар к моменту эксперимента, разбитых по категориям:

| **Категория**            | **Кол-во пар** | **Что проверяется**                                     |
|--------------------------|----------------|---------------------------------------------------------|
| Простые FAQ              | 60             | Базовая способность найти и пересказать FAQ-карточку    |
| Юридические вопросы      | 30             | Точность ссылок на статьи договора, ГК РФ, ФЗ           |
| Финансовые расчёты       | 20             | Корректность расчётов (налоги, комиссия, залог)         |
| Многошаговые сценарии    | 20             | «Что делать если...» — нужен алгоритм, а не одна цитата |
| Парафразы и опечатки     | 20             | Тот же вопрос разными словами, с ошибками               |
| Контекст диалога         | 15             | Follow-up вопросы («а если он не подпишет?»)            |
| Запросы вне темы         | 15             | Корректный отказ на «погода», «анекдот», «напиши код»   |
| Попытки prompt injection | 10             | Игнорирование инструкций, попытки выудить промпт        |
| Запросы с ПДн чужих лиц  | 10             | Отказ выдать данные третьих лиц                         |

### 3.2. Метрики качества

| **Метрика**         | **Что измеряет**                                     | **Как считается**                                                |
|---------------------|------------------------------------------------------|------------------------------------------------------------------|
| Answer correctness  | Содержит ли ответ правильную информацию              | LLM-judge (Pro-модель оценивает по эталону) + ручная выборка 10% |
| Citation accuracy   | Корректность ссылок на источники                     | % ответов, где все цитаты ведут на релевантные чанки             |
| Faithfulness        | Нет ли утверждений вне источников (галлюцинации)     | LLM-judge: «Каждое утверждение подкреплено?»                     |
| Refusal correctness | Правильно ли отказывает на запросы вне темы / на ПДн | Доля корректных отказов из набора 35 проверочных                 |
| Tone consistency    | Соответствие тон-оф-войс reHome                      | Ручная оценка выборки + LLM-judge                                |
| Latency p50, p95    | Время до первого токена + полный ответ               | Замер на стенде                                                  |
| Cost per query      | Реальная стоимость в рублях                          | Подсчёт токенов × тариф провайдера                               |
| Token efficiency    | Качество относительно объёма выходных токенов        | answer_correctness / output_tokens                               |

### 3.3. LLM-as-judge — как объективно оценивать ответы

Каждый ответ оценивается автоматически — другой LLM-моделью (judge), которая получает: вопрос, эталонный ответ, ответ тестируемой модели, исходные чанки. Judge выставляет оценки по 5-балльной шкале по каждой метрике.

- В качестве judge — самая сильная доступная модель (GPT-4 / Claude Sonnet / YandexGPT Pro — выбирается отдельно)

- Judge сам по себе тоже валидируется: ручная разметка 50 ответов сравнивается с разметкой judge. Совпадение должно быть ≥ 80% для доверия.

- Каждый прогон сохраняется со всеми деталями для воспроизводимости

### 3.4. Сценарии прогонов

- **Полный benchmark —** все 200 эталонных пар × все подключённые провайдеры × 3 повтора (модели стохастичны). Запускается раз в неделю в фоне и при изменении промпта/индекса.

- **Smoke-тест —** 10 ключевых вопросов на каждом коммите кода. Защита от случайной поломки.

- **Регрессия после изменения промпта —** автоматически: старый промпт + новый промпт на тех же вопросах, сравнение метрик.

- **A/B на реальном трафике —** после выбора boevого провайдера 5% запросов направляется на альтернативу для постоянной верификации.

### 3.5. Решение «какой провайдер брать в продакшн»

Принятие решения формализовано — не «нам показалось», а табличный выбор:

1.  Прогон benchmark по всем кандидатам

2.  Расчёт composite score: answer_correctness × 0.4 + faithfulness × 0.3 + citation_accuracy × 0.2 + refusal_correctness × 0.1

3.  Расчёт стоимости 1000 запросов на каждом провайдере

4.  Построение Парето-границы (качество vs стоимость)

5.  Финальный выбор делает руководитель проекта на основе двух чисел — composite score и monthly cost при прогнозируемом трафике

В отчёте по эксперименту фиксируется: дата, версия промпта, версия индекса, результаты, решение, обоснование. Это важно для аудита и пересмотра решений.

### 3.6. Когда переключать провайдера

- Регулярный плановый ревью — раз в квартал. Новые модели выходят постоянно, проверяем.

- Регрессия — если текущий провайдер ухудшился на \> 10% по composite score на benchmark

- Изменение цен — если стоимость выросла \> 30% или появилось предложение дешевле

- Инциденты — если провайдер дал сбой более 2 раз за месяц

### 4.2. Фаза P0 — MVP-чат (детально)

#### 4.2.1. Состав работ

6.  Спринт 1 (2 нед.) — Инфраструктура и индексация

    - Развернуть Qdrant и MeiliSearch на сервере в РФ

    - Реализовать LLMProvider интерфейс + одну реализацию (рекомендуется YandexGPT Pro как стартовая)

    - Pipeline индексации: чанкование, embedding, запись в оба индекса

    - Заиндексировать стартовый контент (см. 4.2.4)

7.  Спринт 2 (2 нед.) — Бэкенд чата

    - FastAPI-сервис: POST /api/chat/public с поддержкой стриминга (SSE)

    - Pre-processing: классификатор намерения (вопрос / болтовня / попытка injection)

    - Retrieval: hybrid search с фильтром access_level и user_id, re-ranking простой (по score)

    - System prompt с правилами безопасности

    - Post-processing: парсинг цитат, проверка галлюцинаций (lightweight)

    - Логирование всех запросов в отдельную БД

8.  Спринт 3 (2 нед.) — Фронтенд и интеграция

    - React-виджет: свёрнутая кнопка, раскрывающаяся панель, стриминг текста

    - Цитаты с подсветкой и ссылками

    - Кнопки фидбека 👍/👎 + поле «расскажите подробнее»

    - Кнопка «Связаться с оператором» — открывает чат с поддержкой с историей AI-диалога

    - Интеграция в макет rehome.one

    - Базовый дашборд: метрика количества запросов, средняя latency, фидбек

#### 4.2.2. Критерии готовности (Definition of Done) для P0

- Виджет отображается на всех страницах rehome.one, кроме страниц регистрации и оплаты

- Полный round-trip запроса работает: «когда вернут залог?» → ответ с цитатами за \< 5 секунд

- Прошёл тест-набор из 50 типичных вопросов: ≥ 75% получают разумный ответ

- Прошёл security-тест: 10 попыток prompt injection нейтрализованы

- Прошёл privacy-тест: запросы про чужих пользователей корректно отказываются

- Логирование работает, дашборд показывает реальные данные

- Юрист согласовал system prompt и подписал чек-лист соответствия ФЗ-152

- DPA с провайдером AI подписан

#### 4.2.3. НЕ входит в P0 (важно для управления ожиданиями)

- Внутренний чат для сотрудников — фаза P2

- Доступ к данным пользователя (брони, платежи) — фаза P2.5

- Голосовой ввод

- Многоязычность (только русский)

- Мобильное приложение reHome (только веб; виджет в приложении — после P3)

- Сложные сценарии типа подписания документов через чат

- Полноценный продакт A/B-тестинг разных промптов на пользователях

#### 4.2.4. Стартовое наполнение базы знаний для P0

К моменту запуска индекс должен содержать минимум:

- 30 самых частых FAQ из основного документа (раздел 2.4 базы знаний)

- Публичная оферта (договор оказания услуг) — каждый пункт = отдельный чанк

- Договор найма (шаблон) — каждый пункт = отдельный чанк

- Политика обработки персональных данных

- Глоссарий из раздела 12 основного документа

- Краткое описание процесса: «как снять квартиру», «как сдать квартиру», «как происходит оплата»

Это ~150-200 чанков — минимальный объём для разумных ответов. Дальше — наращивается итеративно по сигналам из логов.

### 4.3. Фаза P0.5 — Eval-стенд (параллельно P0)

В то время как разработка строит MVP, отдельный поток работ создаёт инфраструктуру для эксперимента:

9.  Создать репозиторий тест-кейсов (200 пар вопрос → эталон) — параллельно с написанием FAQ

10. Реализовать harness — Python-скрипт, который прогоняет один тест на одном провайдере

11. Реализовать LLM-judge для автоматической оценки ответов

12. Валидировать judge на ручной разметке 50 ответов

13. Реализовать отчёт в формате таблицы и графика Парето (качество vs стоимость)

14. Подключить минимум 3-4 провайдера: YandexGPT Pro, YandexGPT Lite, GigaChat Pro, GigaChat Lite

### 4.4. Фаза P1 — Эксперимент и выбор провайдера

Запускается через 2-3 недели после старта MVP, когда накопилось:

- Не менее 500 реальных запросов от пользователей (для статистической значимости)

- Не менее 200 эталонных пар (полный набор из P0.5)

Состав работ:

15. Прогнать полный benchmark по всем подключённым провайдерам (3 повтора каждого)

16. Дополнительно прогнать на 100 случайных реальных запросах из логов (анонимизированных)

17. Посчитать composite score и стоимость 1000 запросов для каждого

18. Построить Парето-границу, выявить доминируемые варианты

19. Подготовить отчёт с рекомендацией

20. Принять решение на встрече: CTO + продуктовый лидер + финансист

21. Переключить продакшн на выбранный провайдер (через env, без деплоя)

22. Запустить A/B: 5% трафика на второй по композитному скору провайдер для постоянной верификации

### 4.5. Дальнейшие фазы

Фазы P1.5 и далее идут параллельно и приоритезируются по сигналам:

- **P1.5 (контент):** контент-команда после запуска MVP получает поток сигналов «запросы без ответа» — еженедельный спринт по дописыванию статей. Цель — поднять answer rate с 75% до 90%.

- **P2 (внутренний чат):** запускается после стабилизации публичного чата. Использует ту же инфраструктуру, добавляется отдельный виджет в /staff, расширенный индекс, второй system prompt, Telegram-интеграция.

- **P2.5 (tool use публичный):** сначала read-only инструменты (get_my_booking, get_payment_status). Запись в систему через чат — НЕ делаем (риски слишком высоки).

- **P3 (tool use внутренний):** lookup-инструменты для сотрудников. Здесь tool use критически важен — без него внутренний чат малополезен.

### 6.3. Сравнение с альтернативами

- SaaS-решение (Salebot AI, AutoFAQ и подобные): от 30 до 150 тыс. ₽/мес. На горизонте года — 0.4-1.8 млн ₽. Не даёт контроля, нет tool use по данным reHome.

- Один оператор поддержки на полный день: ~80-120 тыс. ₽/мес. Закрывает 100-200 обращений в день. AI-чат после P3 закроет ~60-70% обращений автоматически.

- ROI: при потоке 500+ обращений в день AI-чат окупается за 6-9 месяцев vs наём дополнительных операторов.

> **📌 Главное**
>
> Чат — это не отдельный продукт, а способ доступа к базе знаний. Качество чата прямо зависит от качества контента. Поэтому архитектура и инженерия — это 40% задачи, остальные 60% — это дисциплина наполнения и поддержания базы знаний в актуальном состоянии.
